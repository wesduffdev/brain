Yes — instinct should be a separate layer of the brain, below reasoning, memory, language, and learned object concepts.

The clean way to think about it:

Instinct = fast, low-latency protective reactions before conscious decision-making.
For your example:

If something is coming toward the being fast and unexpectedly, the being may flinch.

That should not require the being to know the word “danger,” “object,” “threat,” or “impact.” It should be based on sensory patterns:

rapid approach
sudden motion
near body
low prediction confidence
high velocity
unexpected trajectory
possible collision
The being does not need to understand the object. It reacts to the pattern.

Brain architecture with instinct
I would add an Instinct Layer between perception and decision-making.

World / Room Events
        ↓

        ↓
Instinct Layer  ← fast protective reactions
        ↓
Emotion System
        ↓
Prediction / Memory / Learning
        ↓

Current action is interrupted

Instinct should not fully replace safety rules
Even if you want instinct to be learned and not rule-based, you should still keep hard safety boundaries in the code.

Think of it this way:

Neural instinct model:

Safety layer:
prevents impossible, invalid, or unsafe behavior.
Example:

Model predicts: flinch backward
But there is a wall behind the being
Action system changes it to: flinch in place / cover face
This is important because the instinct model should not know every physical constraint.

How instinct differs from emotion
Instinct and emotion are related, but not the same.

Instinct:
fast body reaction

Emotion:
internal affective state

Decision:
chosen action after evaluation
Example:

Something flies toward face

Instinct:
flinch

Emotion:
scared

Decision after flinch:
move away, inspect object, cry/call out, or resume activity
So do not model “flinch” as an emotion. Model it as a reaction.

The being may learn:

This is pre-conceptual.

Why events are better than ticks here
Ticks are not bad. Games use ticks all the time. But if everything depends on ticks, you end up with:

10,000 objects × every tick × every system
That becomes noisy and expensive.

Event-driven means the system reacts only when something meaningful changes:

object appeared
object moved fast
sound occurred
light level changed
object touched being
being changed location
being looked at object
object dropped
object collided
temperature changed
prediction was wrong
This is much closer to how your “brain” model should work.

Important clarification
You may still need a small scheduler, but not a giant global tick brain.

Use events as the core, and use scheduled events only for things that naturally change over time.

Example:

Hunger increasing over time → scheduled need-change event
Object moving → physics/motion event
Light turning off → environment event
Fast object approaching → perception event
So instead of:

Every tick:
  update hunger
  update sleep
  check every object
  check every light
  check every sound
  ask neural net
Use:

At time T:
  emit NeedChanged(hunger +1)

When object velocity crosses threshold:
  emit ApproachingObjectDetected

When sound happens:
  emit SoundDetected

When light changes:
  emit LightChanged
That is the better long-term architecture.

What should still use time?
Some things are time-based:

needs drift
emotion decay
cooldowns
memory consolidation
training jobs
model refresh
But they should emit events, not force all systems to poll.

Example:

NeedScheduler emits:
NeedDriftEvent(hunger, +1)

EmotionDecayScheduler emits:
EmotionDecayEvent(scared, -0.05)

InstinctCooldownScheduler emits:
CooldownExpired(flinch)
That way time exists, but behavior remains event-driven.

# NPC Brain Augmentation Plan: Kafka Event Architecture + Instinct Neural Network

## Purpose

This document is an **augmentation plan** for the existing NPC brain architecture. It does not replace the prior v0/v1 plans. It focuses only on the new direction:

- Add an **instinct brain module**.
- Move selected runtime behavior from tick-driven execution to **Kafka event-driven execution**.
- Decide whether the existing neural network should be extended or whether a new neural network should be added.
- Define a parallelizable implementation plan that can be converted into engineering tickets.

The current system already has a neural network and existing architecture work. This plan assumes those exist and should be enhanced rather than restarted.

---

## Strategic Decision

Use **Kafka as the event backbone** for this next phase.

The goal is not to build a small temporary in-process event bus first. The goal is to introduce the production-shape architecture now so that future work does not require a major messaging refactor.

Kafka should become the durable event layer for:

- perception events
- sensory stimulus events
- instinct reaction events
- neural prediction events
- action interruption events
- state transition events
- training example capture
- model evaluation telemetry

The existing tick system should not be removed blindly. Instead, we should classify which responsibilities should remain time/scheduler-driven and which should become event-driven.

---

## Key Architecture Principle

The brain should not rely on every subsystem polling every tick.

Instead:

```text
Meaningful change occurs
  -> Kafka event is produced
  -> subscribed services react
  -> state changes are persisted
  -> downstream events are emitted
  -> neural training examples are captured
```

Ticks may still exist, but only as a way to generate scheduled events when time itself is the cause of change.

Example:

```text
Old model:
Every tick, hunger checks whether it should increase.

New model:
A scheduler emits NeedDriftRequested or NeedChanged at configured intervals.
```

This keeps time-based behavior tunable without requiring every service to poll constantly.

---

## Kafka Listener Concern

Having many listeners is not inherently a problem for Kafka if the topology is designed correctly.

Kafka is built around topics, partitions, producers, consumers, and consumer groups. The main design concern is not “too many listeners” by itself. The main concerns are:

- too many consumer groups reading the same high-volume topics
- topics with insufficient partitions for expected throughput
- consumers doing heavy synchronous work inside the poll loop
- lack of idempotency when events are replayed
- poor topic boundaries causing every service to consume everything
- unbounded event fan-out
- missing schema/versioning strategy

The desired pattern is:

```text
Many services may consume events,
but each service should consume only the topics it needs,
and consumer groups should be intentionally designed.
```

A listener is acceptable when it has a clear ownership boundary. A listener is a smell when it passively subscribes to broad topics and filters most events locally.

---

## Event-Driven vs Tick-Driven Classification Plan

Before refactoring, create a classification matrix for all existing tick-based behavior.

Each tick-driven responsibility should be classified as one of the following:

| Classification | Meaning | Target Design |
|---|---|---|
| Event-driven | Should only run when something meaningful happens | Kafka event consumer |
| Scheduled event | Time causes the change, but should emit events instead of polling everywhere | Scheduler -> Kafka event |
| Real-time loop | Truly needs frequent simulation updates | Keep local loop, emit summarized events |
| Renderer-only | Visual interpolation or animation detail | Keep in renderer, not brain |
| Training/analytics | Should run async or batch | Kafka consumer or trainer job |

## Examples

| Behavior | Current Style | New Classification | Recommendation |
|---|---|---|---|
| Object appears | Tick scan | Event-driven | Emit `ObjectAppeared` |
| Object moves fast toward being | Tick scan | Event-driven / perception-derived | Emit `ObjectApproached` |
| Sudden sound | Tick scan | Event-driven | Emit `SoundOccurred` |
| Light level changes | Tick scan | Event-driven | Emit `LightChanged` |
| Need slowly increases | Tick update | Scheduled event | Emit `NeedDriftRequested` or `NeedChanged` |
| Emotion decay | Tick update | Scheduled event | Emit `EmotionDecayRequested` |
| Animation frame interpolation | Tick update | Renderer-only | Keep inside PixiJS/renderer |
| Neural training | Tick/batch | Training/analytics | Consume events asynchronously |
| Memory consolidation | Tick/batch | Scheduled/batch event | Emit `MemoryConsolidationRequested` |
| Instinct flinch | Rule/tick | Event-driven neural inference | Consume `ObjectApproached` |

## Proposed New Brain Layer: Instinct Module

The instinct module is an additional neural component. It should sit between perception and higher-level decision-making.

```text
World / Environment Change
  -> Perception Event
  -> Instinct Neural Inference
  -> Instinct Reaction Candidate
  -> Decision / Safety Validation
  -> Action Interruption or State Update
  -> Render Event
  -> Memory + Training Capture
```

Instinct is not the same as emotion.

```text
Instinct = fast reaction
Emotion = internal affective state
Decision = selected action after validation
```

Example:

```text
A fast object moves toward the being unexpectedly.

Instinct:
flinch

Emotion bias:
startled / scared

Decision after reaction:
resume, withdraw, inspect, or move away
```

## Neural Network Strategy

Because an existing neural network already exists, the first task is to evaluate whether instinct should be added as:

1. an extension of the existing model, or
2. a separate specialized neural model.

## Option A: Extend Existing Neural Network

Use this if the current model already handles:

- perception features
- object/action/context inputs
- reaction prediction
- multi-output classification
- event-based inference
- model versioning

Potential benefits:

- fewer model artifacts
- shared feature encoder
- simpler deployment if the existing model-service is already mature

Risks:

- model becomes too broad
- harder to test instinct separately
- harder to iterate on fast reactions without affecting other predictions
- instinct needs low-latency behavior that may not match other model workloads

## Option B: Add a New Instinct Neural Network

Use this if instinct is meaningfully different from the current model.

Recommended default: **add a separate instinct model** unless the existing model is already architected for multiple heads and low-latency event inference.

Benefits:

- clean module boundary
- easier training data design
- easier shadow-mode rollout
- easier model evaluation
- lower blast radius
- easier to replace later

Risks:

- additional model artifact
- additional inference pathway
- more deployment/versioning concerns

## Recommended Decision

Start with a short architectural spike:

```text
Spike: Determine whether instinct should be a new model or a new head on the existing model.
```

The spike should inspect:

- current model input features
- current output labels
- current training data shape
- current inference latency
- current model artifact/versioning
- current service ownership
- whether the existing model supports multi-task heads

Expected outcome:

```text
Decision record:
- Extend existing model
or
- Add new instinct model
```

If uncertain, choose a separate instinct model first because it is easier to isolate and test.

## Preventing Too Many Listener Problems

Kafka can handle many consumers, but the architecture should avoid uncontrolled fan-out.

Use these controls:

## 1. Topic Ownership

Each topic must have an owner.

```text
Topic owner decides schema, retention, and compatibility rules.
```

## 2. Narrow Topic Subscriptions

A service should only subscribe to topics it needs.

Avoid:

```text
All services subscribe to npc.all.events
```

Prefer:

```text
Instinct service subscribes to npc.perception.events
Renderer subscribes to npc.render.events
Training service subscribes to selected event topics
```

## 3. Schema Registry / Versioning

Use a schema strategy early.

Recommended:

```text
JSON Schema or Avro with schema registry
```

For early implementation, JSON Schema is easier. If the team already has Avro/Schema Registry standards, use that.

## 4. Idempotent Consumers

All consumers should handle duplicate events safely.

Use:

```text
event_id as idempotency key
```

## 5. Dead Letter Topics

Create dead letter topics for failed event handling.

Examples:

```text
npc.perception.events.dlq
npc.instinct.reactions.dlq
npc.training.examples.dlq
```

## 6. Backpressure and Heavy Work

Consumers should not do expensive synchronous work in tight poll loops.

For model inference:

```text
consume event
batch if useful
run inference
produce prediction event
commit offset only after success
```

For training:

```text
consume events or read Postgres asynchronously
batch into training examples
train in separate job
```

# NPC Brain Augmentation Plan: Kafka Event Architecture + Instinct Neural Network

## Purpose

This document is an **augmentation plan** for the existing NPC brain architecture. It does not replace the prior v0/v1 plans. It focuses only on the new direction:

- Add an **instinct brain module**.
- Move selected runtime behavior from tick-driven execution to **Kafka event-driven execution**.
- Decide whether the existing neural network should be extended or whether a new neural network should be added.
- Define a parallelizable implementation plan that can be converted into engineering tickets.

The current system already has a neural network and existing architecture work. This plan assumes those exist and should be enhanced rather than restarted.

---

## Strategic Decision

Use **Kafka as the event backbone** for this next phase.

The goal is not to build a small temporary in-process event bus first. The goal is to introduce the production-shape architecture now so that future work does not require a major messaging refactor.

Kafka should become the durable event layer for:

- perception events
- sensory stimulus events
- instinct reaction events
- neural prediction events
- action interruption events
- state transition events
- training example capture
- model evaluation telemetry

The existing tick system should not be removed blindly. Instead, we should classify which responsibilities should remain time/scheduler-driven and which should become event-driven.

---

## Key Architecture Principle

The brain should not rely on every subsystem polling every tick.

Instead:

```text
Meaningful change occurs
  -> Kafka event is produced
  -> subscribed services react
  -> state changes are persisted
  -> downstream events are emitted
  -> neural training examples are captured
```

Ticks may still exist, but only as a way to generate scheduled events when time itself is the cause of change.

Example:

```text
Old model:
Every tick, hunger checks whether it should increase.

New model:
A scheduler emits NeedDriftRequested or NeedChanged at configured intervals.
```

This keeps time-based behavior tunable without requiring every service to poll constantly.

---

## Kafka Listener Concern

Having many listeners is not inherently a problem for Kafka if the topology is designed correctly.

Kafka is built around topics, partitions, producers, consumers, and consumer groups. The main design concern is not “too many listeners” by itself. The main concerns are:

- too many consumer groups reading the same high-volume topics
- topics with insufficient partitions for expected throughput
- consumers doing heavy synchronous work inside the poll loop
- lack of idempotency when events are replayed
- poor topic boundaries causing every service to consume everything
- unbounded event fan-out
- missing schema/versioning strategy

The desired pattern is:

```text
Many services may consume events,
but each service should consume only the topics it needs,
and consumer groups should be intentionally designed.
```

A listener is acceptable when it has a clear ownership boundary. A listener is a smell when it passively subscribes to broad topics and filters most events locally.

---

## Event-Driven vs Tick-Driven Classification Plan

Before refactoring, create a classification matrix for all existing tick-based behavior.

Each tick-driven responsibility should be classified as one of the following:

| Classification | Meaning | Target Design |
|---|---|---|
| Event-driven | Should only run when something meaningful happens | Kafka event consumer |
| Scheduled event | Time causes the change, but should emit events instead of polling everywhere | Scheduler -> Kafka event |
| Real-time loop | Truly needs frequent simulation updates | Keep local loop, emit summarized events |
| Renderer-only | Visual interpolation or animation detail | Keep in renderer, not brain |
| Training/analytics | Should run async or batch | Kafka consumer or trainer job |

## Examples

| Behavior | Current Style | New Classification | Recommendation |
|---|---|---|---|
| Object appears | Tick scan | Event-driven | Emit `ObjectAppeared` |
| Object moves fast toward being | Tick scan | Event-driven / perception-derived | Emit `ObjectApproached` |
| Sudden sound | Tick scan | Event-driven | Emit `SoundOccurred` |
| Light level changes | Tick scan | Event-driven | Emit `LightChanged` |
| Need slowly increases | Tick update | Scheduled event | Emit `NeedDriftRequested` or `NeedChanged` |
| Emotion decay | Tick update | Scheduled event | Emit `EmotionDecayRequested` |
| Animation frame interpolation | Tick update | Renderer-only | Keep inside PixiJS/renderer |
| Neural training | Tick/batch | Training/analytics | Consume events asynchronously |
| Memory consolidation | Tick/batch | Scheduled/batch event | Emit `MemoryConsolidationRequested` |
| Instinct flinch | Rule/tick | Event-driven neural inference | Consume `ObjectApproached` |

## Proposed New Brain Layer: Instinct Module

The instinct module is an additional neural component. It should sit between perception and higher-level decision-making.

```text
World / Environment Change
  -> Perception Event
  -> Instinct Neural Inference
  -> Instinct Reaction Candidate
  -> Decision / Safety Validation
  -> Action Interruption or State Update
  -> Render Event
  -> Memory + Training Capture
```

Instinct is not the same as emotion.

```text
Instinct = fast reaction
Emotion = internal affective state
Decision = selected action after validation
```

Example:

```text
A fast object moves toward the being unexpectedly.

Instinct:
flinch

Emotion bias:
startled / scared

Decision after reaction:
resume, withdraw, inspect, or move away
```

## Neural Network Strategy

Because an existing neural network already exists, the first task is to evaluate whether instinct should be added as:

1. an extension of the existing model, or
2. a separate specialized neural model.

## Option A: Extend Existing Neural Network

Use this if the current model already handles:

- perception features
- object/action/context inputs
- reaction prediction
- multi-output classification
- event-based inference
- model versioning

Potential benefits:

- fewer model artifacts
- shared feature encoder
- simpler deployment if the existing model-service is already mature

Risks:

- model becomes too broad
- harder to test instinct separately
- harder to iterate on fast reactions without affecting other predictions
- instinct needs low-latency behavior that may not match other model workloads

## Option B: Add a New Instinct Neural Network

Use this if instinct is meaningfully different from the current model.

Recommended default: **add a separate instinct model** unless the existing model is already architected for multiple heads and low-latency event inference.

Benefits:

- clean module boundary
- easier training data design
- easier shadow-mode rollout
- easier model evaluation
- lower blast radius
- easier to replace later

Risks:

- additional model artifact
- additional inference pathway
- more deployment/versioning concerns

## Recommended Decision

Start with a short architectural spike:

```text
Spike: Determine whether instinct should be a new model or a new head on the existing model.
```

The spike should inspect:

- current model input features
- current output labels
- current training data shape
- current inference latency
- current model artifact/versioning
- current service ownership
- whether the existing model supports multi-task heads

Expected outcome:

```text
Decision record:
- Extend existing model
or
- Add new instinct model
```

If uncertain, choose a separate instinct model first because it is easier to isolate and test.

## Kafka Consumer Group Guidance

Use consumer groups by service responsibility.

Example:


## Preventing Too Many Listener Problems

Kafka can handle many consumers, but the architecture should avoid uncontrolled fan-out.

Use these controls:

## 1. Topic Ownership

Each topic must have an owner.

```text
Topic owner decides schema, retention, and compatibility rules.
```

## 2. Narrow Topic Subscriptions

A service should only subscribe to topics it needs.

Avoid:

```text
All services subscribe to npc.all.events
```

Prefer:

```text
Instinct service subscribes to npc.perception.events
Renderer subscribes to npc.render.events
Training service subscribes to selected event topics
```

## 3. Schema Registry / Versioning

Use a schema strategy early.

Recommended:

```text
JSON Schema or Avro with schema registry
```

For early implementation, JSON Schema is easier. If the team already has Avro/Schema Registry standards, use that.

## 4. Idempotent Consumers

All consumers should handle duplicate events safely.

Use:

```text
event_id as idempotency key
```

## 5. Dead Letter Topics

Create dead letter topics for failed event handling.

Examples:

```text
npc.perception.events.dlq
npc.instinct.reactions.dlq
npc.training.examples.dlq
```

## 6. Backpressure and Heavy Work

Consumers should not do expensive synchronous work in tight poll loops.

For model inference:

```text
consume event
batch if useful
run inference
produce prediction event
commit offset only after success
```

For training:

```text
consume events or read Postgres asynchronously
batch into training examples
train in separate job
```

## Rollout Strategy

Use staged rollout to reduce risk.

## Phase 1: Shadow Mode

Kafka events are produced and consumed, but instinct reactions do not interrupt behavior.

```text
Perception event -> instinct prediction -> log only
```

Acceptance criteria:

```text
predictions are created
predictions are persisted
consumer lag is acceptable
event chains are traceable
no behavior changes yet
```

## Phase 2: Visual-Only Mode

Instinct reactions may trigger renderer animations, but do not interrupt core action state.

```text
ObjectApproached -> flinch animation only
```

Acceptance criteria:

```text
renderer reacts correctly
no invalid state transitions
reaction cooldowns work
```

## Phase 3: Controlled Interruption Mode

High-confidence reactions may interrupt interruptible actions.

```text
ObjectApproached -> flinch -> interrupt current action if allowed
```

Acceptance criteria:

```text
only allowed actions are interrupted
state remains valid
reaction is persisted
training example is generated
```

## Phase 4: Adaptive Training Mode

Event chains are used to improve the instinct model.

```text
Kafka events -> training examples -> trainer -> new model version
```

Acceptance criteria:

```text
training job can reproduce model artifact
model metrics are stored
new model can be deployed by version
rollback is possible
```
